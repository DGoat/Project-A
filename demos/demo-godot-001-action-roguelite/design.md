# 设计文档

## 当前主题方向：玩具修理屋

一句话概念：

```text
夜晚的修理屋里，坏掉的玩具王国醒了。你是小小修理师，进入玩具箱王国，把发怒、断线、生锈、遗忘的玩具们修好。
```

当前 Demo 的主题替换原则：

| 通用动作 Roguelite | 玩具修理屋主题 |
|---|---|
| 玩家 | 小小修理师 / 发条修理精灵 |
| 敌人 | 失控玩具 |
| 攻击 | 修理打击 |
| Dash | 发条冲刺 |
| 祝福 | 修理灵感 |
| 击杀 | 修复完成 |
| 房间 | 玩具箱区域 |
| 胜利 | 黎明前修好了所有玩具 |
| 失败 | 小灯熄灭，被送回修理台 |

优先构筑路线：

- 发条：攻速 / 连击 / 过载。
- 缝补：回血 / 护盾 / 容错。
- 积木：范围 / 击退 / 空间控制。
- 灯光：净化 / 持续修复。
- 胶水：控制 / 黏住 / 区域限制。

## 高概念（High Concept）

`demo-godot-001-action-roguelite` 是一个小体量俯视角 2D 战斗原型，核心关注 **动作手感（Action Feel）** 与 **局内构筑选择（Build Choice）**。

目标不是堆内容量，而是验证一条可复用的 Godot 2D 生产路径：

```text
AI 辅助设计 -> 小型可玩原型 -> 数据驱动调参 -> 记录生成过程
```

## 设计支柱（Design Pillars）

1. 战斗信息清晰、可读
2. 操作简单直接
3. 战斗后提供构筑选择
4. 小房间、短流程
5. 祝福（Blessing）数据驱动，方便调参

## 玩家（Player）

### 初始数值（Stats）

| 数值 | 初始值 |
|---|---:|
| 最大生命值 Max HP | 100 |
| 移动速度 Move Speed | 240 |
| 闪避速度 Dash Speed | 620 |
| 闪避持续时间 Dash Duration | 0.16 |
| 闪避冷却 Dash Cooldown | 0.55 |
| 攻击伤害 Attack Damage | 20 |
| 攻击冷却 Attack Cooldown | 0.35 |
| 攻击范围 Attack Range | 56 |

### 玩家动作（Actions）

- 8 方向移动
- 按当前移动方向闪避（Dash）
- 基础近战攻击
- 受伤
- 死亡并重开

## 敌人（Enemies）

### 近战追击怪（Melee Chaser）

- 持续向玩家移动
- 接触造成伤害
- 生命值较低
- 教玩家保持距离、利用闪避调整位置

### 远程射手（Ranged Shooter）

- 与玩家保持距离
- 朝玩家发射弹体（Projectile）
- 移动速度较慢
- 迫使玩家移动、走位、优先击杀

## 房间（Rooms）

| 房间 | 敌人配置 | 设计目的 |
|---|---|---|
| 1 | 2 个近战追击怪 | 学习移动和攻击 |
| 2 | 2 个近战追击怪 + 1 个远程射手 | 引入目标优先级 |
| 3 | 1 个精英近战怪 + 2 个远程射手 | 检验构筑和操作 |

## 祝福（Blessings）

玩家在清理第 1、2 个房间后，从 3 个祝福中选择 1 个。

初始祝福池：

| Id | 名称 | 效果 |
|---|---|---|
| attack_up | Sharpened Edge | 攻击伤害 +20% |
| attack_speed_up | Quick Hands | 攻击冷却 -15% |
| dash_strike | Wind Step | 闪避后下一次攻击伤害 +50% |
| kill_heal | Blood Warmth | 击杀敌人后回复 8 点生命 |
| burn_attack | Ember Mark | 攻击附加燃烧伤害 |
| range_up | Long Reach | 攻击范围 +20% |

## 标杆游戏拆解（Benchmark Analysis）

### Hollow Knight / Silksong

可借鉴点：

- 战斗可读性来自清晰的前摇（Wind-up）、生效帧（Active Frames）和后摇（Recovery）。
- 玩家移动应该精准、轻快，而不是笨重。
- 单个敌人可以简单，但多个敌人组合后要产生压力。
- 命中反馈需要立刻发生：闪白、击退、短暂停顿（Hit Stop）、音效。
- 第一版不做探索大地图，但房间布局和敌人摆放仍要有节奏。

本 Demo 借鉴：

- 可读的敌人行为
- 基于距离和站位的近战战斗
- 闪避和走位压力
- 少量但职责清晰的敌人类型

暂不借鉴：

- 类银河恶魔城大地图
- 精密平台跳跃
- 大型 Boss 招式组
- 复杂动画需求

### Slay the Spire

可借鉴点：

- 三选一奖励最有价值的地方，不是给玩家加数值，而是暗示不同构筑方向。
- 好选择不一定是纯增强，也可以带来取舍和协同（Synergy）。
- 一局游戏让人记住，往往来自前几次选择塑造出的构筑路线。
- 奖励文本必须清楚，玩家要在几秒内理解效果。

本 Demo 借鉴：

- 三选一奖励界面
- 能定义构筑方向的祝福
- 简短清晰的效果说明
- 数据驱动的祝福池

暂不借鉴：

- 卡组管理
- 大量卡牌/遗物池
- 复杂敌人意图 UI
- 长周期数值平衡

### 暖雪（Warm Snow）

可借鉴点：

- 快节奏房间战斗适合搭配强烈流派幻想和构筑协同。
- 核心攻击可以简单，只要构筑效果能改变攻击节奏和输出方式。
- 轻量敌人 + 数量/组合升级，就能制造压力，不一定需要复杂 AI。
- 当效果叠加时，视觉清晰度非常重要。

本 Demo 借鉴：

- 2D 房间战斗
- 清房 -> 奖励 -> 下一房间的快节奏循环
- 燃烧、攻击范围、攻速、击杀回血等攻击修饰
- 短流程 Run 结构

暂不借鉴：

- 大型门派/圣物系统
- 大量飞剑/飞行武器
- 高密度 VFX
- 局外成长（Meta Progression）

### Hades

可借鉴点：

- 房间节奏很重要：战斗爆发、奖励暂停、进入下一房间的期待感。
- 祝福（Boon）好玩，是因为它们改造核心动作，而不只是增加数值。
- 敌人应该迫使玩家移动，而不是原地换血。
- 如果每个房间目标清晰，Run 可以很短。

本 Demo 借鉴：

- 清房奖励节奏
- 祝福改造核心动作
- 近战 + 远程敌人制造走位压力
- 三房间短流程

暂不借鉴：

- 叙事基地
- 大量角色关系
- 多武器系统
- 程序化房间图

## 标杆拆解后的设计方向

本 Demo 应该发展成：

```text
短房间、强可读敌人、构筑型祝福驱动的俯视角 2D 动作 Roguelite。
```

本 Demo 不应该同时变成：

```text
类银河恶魔城 + 平台跳跃 + 完整 Roguelite + 叙事基地。
```

## 第一版成功标准（First Prototype Success Criteria）

第一版可玩原型成功的标准：

1. 移动响应及时。
2. 闪避能帮助玩家规避危险和重新站位。
3. 普攻有清晰的范围和节奏。
4. 近战敌人与远程敌人带来不同压力。
5. 清房后有明确的暂停和奖励节奏。
6. 玩家能在 3 秒内理解祝福效果。
7. 至少 2 个祝福能明显改变战斗体验。
8. 一局完整流程控制在 2-4 分钟。

## 核心手感目标（Core Feel Targets）

| 模块 | 目标 |
|---|---|
| 移动 | 快速、直接，不加入复杂加速度负担 |
| 闪避 | 短爆发、用于战术脱离和调整位置 |
| 攻击 | 快速近战戳击，范围清晰 |
| 敌人伤害 | 可规避，不应像随机扣血 |
| 房间时长 | 20-45 秒 |
| 奖励选择 | 短暂停顿、效果清楚、立即生效 |
| Run 长度 | 第一版 3 个房间 |

## 原型设计风险（Prototype Design Risks）

| 风险 | 应对方式 |
|---|---|
| 战斗发飘 | 添加命中闪白、Hit Stop、击退、攻击范围提示 |
| 祝福像隐形数值 | 优先做可见效果：范围、燃烧、闪避强化、剑气 |
| 房间重复 | 先调整敌人组合和出生点，再加新机制 |
| 范围膨胀 | 第一版锁定 3 房间、6 祝福 |
| 太像标杆游戏 | 借鉴机制，不复制题材、角色和内容表达 |

## 构筑路线目标（Build Route Targets）

### 路线 A：燃烧 / 持续伤害（Burn / Damage Over Time）

幻想：

```text
持续压制敌人，叠加延迟伤害，通过进攻节奏取胜。
```

示例祝福：

- 攻击附加燃烧
- 燃烧持续时间增加
- 燃烧中的敌人受到更多伤害
- 击杀燃烧中的敌人回复生命

### 路线 B：闪避强化 / 爆发（Dash Strike / Burst）

幻想：

```text
用闪避穿过危险，再打出一次强化攻击。
```

示例祝福：

- 闪避后下一次攻击强化
- 击杀后降低闪避冷却
- 闪避留下伤害轨迹
- 强化攻击击退敌人

### 路线 C：攻击距离 / 安全站位（Reach / Safe Spacing）

幻想：

```text
控制距离，安全消耗，避免被包围。
```

示例祝福：

- 攻击范围增加
- 攻击扇形范围变宽
- 命中敌人后使其减速
- 每第 3 次攻击释放剑气

## AI 辅助生产机会（AI-Assisted Production Opportunities）

AI 适合协助生成：

- 按构筑路线生成祝福候选
- 敌人行为卡（Enemy Behavior Cards）
- 房间遭遇表（Encounter Tables）
- 数值调参表
- 测试用例
- 每轮迭代后的 Patch Notes
- 可复现的 Prompt 记录

人需要最终判断：

- 手感是否舒服
- 难度是否合适
- 题材与美术方向
- 哪些效果真正好玩
- 哪条构筑路线值得继续扩展

## 玩家幻想（Player Fantasy）

玩家幻想是：

```text
我进入危险房间，读懂敌人动作，在正确时机闪避，反击，然后选择一个祝福，让下一场战斗变得不一样。
```

## 开发关键节点（Milestones）

### M0：项目骨架完成

目标：Godot 项目能被打开，基础文件结构完整。

完成标准：

- `project.godot` 存在
- `Main.tscn` 存在
- 玩家、敌人、子弹、祝福数据文件存在
- README、设计文档、生成记录存在

当前状态：已完成。

### M1：最小可运行闭环（First Playable）

目标：玩家可以完整跑完一局 3 房间流程。

完成标准：

- 玩家可移动、闪避、攻击
- 敌人可追击/射击
- 玩家和敌人能互相造成伤害
- 清房后出现三选一祝福
- 祝福能改变至少一个战斗参数
- 清理第 3 房间后胜利
- 玩家死亡后可按 `R` 重开
- 本地 Godot 4.x 可运行，无 API Key

当前状态：代码骨架已完成，需要实际运行验证和修 bug。

### M2：基础手感可接受

目标：从“能跑”变成“能打”。

完成标准：

- 攻击范围清晰可见
- 命中有闪白或反馈
- 受击有反馈
- 玩家受伤后有短暂无敌
- 闪避手感不拖沓
- 敌人接触伤害不造成连续秒杀
- 房间 1 能稳定教学移动/攻击/闪避

### M3：构筑效果可感知

目标：祝福不只是数字变化，而是能让战斗方式发生变化。

完成标准：

- 至少 2 条构筑路线初步成立：
  - 燃烧 / 持续伤害
  - 闪避强化 / 爆发
- 至少 4 个祝福有明显可见效果
- 祝福选择后，下一房间能立刻感到差异
- 祝福文本清楚，3 秒内能理解

### M4：小型垂直切片（Vertical Slice）

目标：形成一个可展示的小 Demo。

完成标准：

- 3-5 个房间
- 1 个小 Boss 或精英战
- 10-12 个祝福
- 简单开始/胜利/失败界面
- 基础音效
- 基础粒子或打击反馈
- README 中有明确运行方式和截图/GIF

### M5：差异化验证

目标：从实际游玩中选出一个值得放大的差异点。

完成标准：

- 至少 3 次自测记录
- 至少 2 位外部玩家反馈
- 记录最爽瞬间、最无聊瞬间、最强祝福、最弱祝福
- 选定一个核心差异化方向进入下一轮设计

候选差异化方向：

- 闪避后构筑：围绕 Dash Timing 形成高风险高收益
- 燃烧连锁：让 DoT、击杀、扩散形成连锁爽点
- 敌人读招：把敌人前摇、破绽、反击做成核心
- 房间事件选择：战斗后不只选祝福，也选择风险/奖励

## 差异化考虑时机

当前阶段只定义差异化边界：

- 不做 Hollow Knight 克隆
- 不做 Slay the Spire 克隆
- 不做 Hades 克隆
- 不做 Warm Snow 克隆
- 不在第一版追求复杂原创系统

最小 Demo 完成后，再根据实际手感选择核心差异化。

原则：

```text
先让基础动作 Roguelite 闭环成立，再从最有趣的手感或构筑苗头里长出差异化。
```

## 胜负条件（Win/Lose）

- 胜利：清理第 3 个房间
- 失败：玩家生命值归 0
- 重开：按 `R`
