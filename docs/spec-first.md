# Spec 先行规范（Spec First）

## 核心原则

中、大型改动必须先产出 Spec，再动手写码。

目标：

```text
想清楚再动手，避免边写边扩、边扩边乱。
```

Spec 用来明确：

- 要解决什么问题
- 不解决什么问题
- 改动范围有多大
- 会影响哪些文件和系统
- 怎么验收
- 风险是什么
- 如果做错了如何回退

---

## 什么时候必须先写 Spec

以下情况必须先写 Spec：

### 1. 新系统

例如：

- 键位配置系统
- 祝福系统重构
- 敌人系统
- 房间系统
- 存档系统
- 设置界面
- 战斗结算系统

### 2. 架构变化

例如：

- 从硬编码改为数据驱动
- 引入统一配置加载
- 引入 UI 管理层
- 引入事件总线
- 引入状态机
- 引入资源管理约定

### 3. 跨多个文件的改动

例如：

- 同时修改场景、脚本、配置、文档
- 改动会影响玩家、敌人、UI、数据文件之间的关系

### 4. 会影响玩法判断的改动

例如：

- 改攻击节奏
- 改闪避机制
- 改敌人伤害模型
- 改祝福选择规则
- 改房间节奏
- 改死亡/胜利流程

### 5. 节点升级

进入新开发节点前，必须先写或更新 Spec：

- M1 -> M2
- M2 -> M3
- M3 -> M4
- M4 -> M5

---

## 哪些改动可以不写 Spec

以下小改动可以直接做，但仍需记录到开发日志：

- 文案修正
- 文档补充
- 注释/格式修正
- 明确的小 bug 修复
- 单个配置值微调
- 单文件内的低风险修复
- 已有 Spec 范围内的小任务

如果不确定是否需要 Spec，默认先写简版 Spec。

---

## Spec 模板

```md
# Spec: 标题

## 背景

为什么要做这个改动？当前问题是什么？

## 目标

- 目标 1
- 目标 2

## 非目标

- 本次不做什么
- 明确不解决什么问题

## 范围

涉及哪些系统、文件、场景、数据。

## 设计方案

说明核心方案、流程、交互和数据流。

## 数据结构 / 配置

如果涉及数据驱动，写清字段和示例。

## 涉及文件

| 文件 | 改动 |
|---|---|
| `path/to/file` | 说明改动 |

## 验收标准

- [ ] 标准 1
- [ ] 标准 2

## 风险

| 风险 | 影响 | 应对 |
|---|---|---|
| 风险 1 | 影响 | 应对 |

## 实施步骤

1. 步骤 1
2. 步骤 2
3. 步骤 3

## 回滚方案

如果失败，如何撤回或降级。

## 日志要求

完成后需要更新：

- `dev-log.md`
- `issue-tracker.md`
- 相关设计文档
```

## 文档存放规则

Demo 内的 Spec 放在：

```text
demos/<demo-id>/specs/
```

命名规则：

```text
spec-<编号>-<主题>.md
```

示例：

```text
spec-001-controls-config.md
spec-002-hit-feedback.md
spec-003-blessing-system.md
```

仓库级 Spec 放在：

```text
docs/specs/
```

## 执行约定

每次中、大型改动流程：

```text
提出需求
-> 写 Spec
-> 用户确认
-> 实施
-> 验证
-> 更新日志和问题追踪
-> 询问是否同步远程 Git
```

注意：

- 未确认 Spec 前，不进入实现。
- 实施过程中如果发现范围变化，需要先更新 Spec。
- Spec 不是一次性文档，可以随讨论迭代。
